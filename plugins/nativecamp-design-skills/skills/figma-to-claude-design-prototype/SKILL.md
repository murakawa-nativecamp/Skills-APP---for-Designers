# Figma → Claude Design プロトタイプ変換スキル

## 目的
Figmaで完成済みのデザイン（静的な画面群）を、**Claude Design 上で実際にタップ・入力できる
インタラクティブプロトタイプ**に変換する。入力エラーの出方・フォーカス時の見た目・
画面遷移の流れを、実機相当の操作感で確認できるようにする。

## 大原則（ガードレール）※重要
1. **完全再現のみ**。Claude Design にデザインのアレンジ・改善提案・「より良い」配色や
   レイアウトの提案を**一切させない**。Figmaのスクリーンショットが唯一の正。
2. **.fig エクスポートは使わない**。コンポーネントがデフォルト状態に戻り、
   作り込んだ状態違い（エラー表示・入力済み等）が失われるため。
   必ず Figma コネクタ（MCP）でメタデータとスクリーンショットを取得する。
3. **Claude Design のログイン画面が出たら、認証情報を入力せず即中断**して
   ユーザーに報告する（ブラウザでログイン済みであることが前提）。
4. **Claude Design の UI が想定と違う・操作対象が見つからない場合は、推測で操作しない**。
   スクリーンショットを撮ってユーザーに報告し、指示を待つ。
5. 特定案件に依存しない。Figma の fileKey / node-id はユーザーから受け取る
   （このスキルに直書きしない）。

## 入力（ユーザーから受け取るもの）
- Figma のファイル URL（`?node-id=...` 付き。対象ページまたは対象フレーム群を指す）
- （任意）対象範囲の指定（例:「決済フローの画面だけ」）

## ワークフロー

### 手順1: Figma からフレーム構造を取得する
- Figma コネクタの `get_metadata` で対象ページの**全フレーム構造**（フレーム名・ID・サイズ）を取得する。
- fileKey は URL から抽出（`/design/<fileKey>/...`）。
- ※ .fig エクスポート経由は禁止（大原則2）。

### 手順2: 「画面 × 状態」のマッピング表を作る
- フレーム名の命名規則は `references/frame-naming.md` を参照。
- 末尾サフィックス `_Default` / `_Focus_◯◯` / `_Filled_◯◯` / `_Error_◯◯` / `_Empty` を持つ
  フレーム群は**同一画面の状態違い**として **1画面に統合**する。
- 出力イメージ:

  | 画面 | 状態フレーム | 実装方針 |
  |---|---|---|
  | Login | Login_Default / Login_Focus_Email / Login_Error_Email / Login_Empty | 1画面・操作で状態切替 |
  | Payment | Payment_Default / Payment_Filled_PwVisible ... | 1画面・操作で状態切替 |

- Action Sheet 系・`block_sheet` 系のフレームは**オーバーレイ**として親画面に紐付ける。

### 手順3: スクリーンショットと数値を取得する ※見た目の正
- 全フレームのスクリーンショットを **PNG で取得・保存**する。**ファイル名 = フレーム名**にする
  （Claude Design 側でどの画像がどの画面・状態かを特定する鍵になるため）。
- これがレンダリング済みの**「見た目の正」**となる。
- 主要画面（各画面の `_Default` と、色・文字が多い画面）は `get_design_context` で
  **色コード・フォント（ファミリー/ウェイト/サイズ）・余白の数値**も取得しておく。

### 手順4: Claude Design でビルドする
1. ブラウザで claude.ai の Claude Design を開く（**ログイン画面が出たら中断**・大原則3）。
2. 新規プロジェクトを作成する。
3. 手順3の **全PNGをアップロード**する。
4. `references/build-prompt-template.md` の雛形に、手順2のマッピング表と手順3の数値を
   埋め込んだ**ビルドプロンプト**を作成して送信する。

### 手順5: 検証ループ（最大3周）
- 生成されたプロトタイプを**実際に操作して**検証する。
  チェック項目は `references/verification-checklist.md` を参照。
- 各画面のスクリーンショットを撮り、Figma のスクショ（手順3）と並べて
  **文言・色・余白・フォント**の差分を確認する。
- 差分があれば**画面名（フレーム名）つき**で Claude Design に修正指示を送る。
  例: 「`Payment_Error_CardNumber` と比べて、エラーメッセージの色が違う。`#F0295D` にして」
- **検証ループは最大3周**。3周で解消しない差分は
  `references/verification-checklist.md` の記録フォーマットで一覧化してユーザーに報告する。

## 落とし穴・注意点
- **スクショのファイル名を変えない**。Claude Design への指示は全てフレーム名で行うため、
  名前がずれると修正指示が伝わらない。
- 状態違いフレームを**別画面として実装させない**（手順2の統合を必ずプロンプトに明記）。
- Claude Design が「改善案」を混ぜてくることがある → ビルドプロンプトの冒頭宣言
  （完全再現のみ・アレンジ禁止）を必ず含め、検証で見つけたら差し戻す。
- 画面数が多い場合も、ビルドプロンプトは**1回で全画面分**を送る（分割すると
  画面間遷移の整合性が崩れやすい）。

## 完了報告に含めるもの
- 画面 × 状態のマッピング表（手順2）
- Claude Design プロジェクトの URL
- 検証結果（チェックリストの合否、残存差分の一覧）
- 中断した場合はその理由とスクリーンショット

## Cowork での依頼の出し方（例）
- 「この Figma をプロトタイプ化して動作確認したい: `https://www.figma.com/design/<fileKey>/...?node-id=...`」
- 「決済フローのエラー表示を Claude Design で試せるようにして」
- 「この画面たちを触れるようにして。フォーカスと画面遷移を確認したい」
